### [BUG-1] 얼굴 인증이 무조건 통과됨

**파일:** `src/pages/car-faceauth/CarFaceAuthPage.jsx:140-158`

**실제 코드:**
```js
// line 140 - 1차 verifyFace 호출 (비동기, await 없음)
verifyFace(licenseImage, selfieData)
  .then((result) => {
    if (result.verified) setStep('success')
    else { setFailMsg('동일인이 아닌 것 같아요.'); setStep('fail') }
  })
  .catch((e) => { setFailMsg(e.message); setStep('fail') })

// line 152 - 결과 기다리지 않고 즉시 성공 처리
setStep('success')   // ← 이 줄이 문제

// line 153 - 동일한 verifyFace를 또 한번 중복 호출
verifyFace(licenseImage, selfieData)
  .then((result) => {
    if (result.verified) setStep('success')
    else { setFailMsg('동일인이 아닌 것 같아요.'); setStep('fail') }
  })
  .catch((e) => { setFailMsg(e.message); setStep('fail') })
```

**원인:**
`verifyFace()`는 `async` 함수이므로 Promise를 반환한다. `await` 없이 호출하면 비동기로 실행되고 JS 엔진은 즉시 다음 줄로 넘어간다. 따라서 line 152의 `setStep('success')`가 인증 결과와 무관하게 **항상 먼저 실행**된다.

추가로 동일한 `verifyFace()` 호출이 두 번 있어 AI 서버에 같은 요청이 2번 전송된다.

**결과:**
- 얼굴 인증 실패해도 무조건 `success` 페이지로 이동
- 얼굴 인증 기능이 사실상 무력화 — 누구든 차를 빌릴 수 있음
- AI 서버 불필요한 2배 부하

**해결방안:**
```js
// line 140-158 전체를 아래로 교체

setStep('processing')

try {
  const result = await verifyFace(licenseImage, selfieData)
  if (result.verified) {
    setStep('success')
  } else {
    setFailMsg('동일인이 아닌 것 같아요. 다시 시도해 주세요.')
    setStep('fail')
  }
} catch (e) {
  console.error('[FaceAuth] 인증 실패:', e)
  setFailMsg(e.message || '인증 중 오류가 발생했어요.')
  setStep('fail')
}
```
단, 이 useEffect는 이미 async context가 아니므로 useEffect 콜백 자체를 `async` IIFE로 감싸거나, 별도 async 함수를 정의해서 호출해야 한다.

### [BUG-2] DIDCardPage에 선언되지 않은 변수 사용 (ReferenceError)

**파일:** `src/pages/did-auth/DIDCardPage.jsx:30, 39`

**실제 코드:**
```js
const copyDID = () => {
  try {
    const ta = document.createElement('textarea')
    ta.value = docId        // ← line 30: docId 미선언
    // ...
    if (ok) {
      setCopied(true)       // ← line 39: setCopied 미선언
      setTimeout(() => setCopied(false), 2000)
    }
  } catch (e) {}
}
```

컴포넌트 상단에 `docId`, `setCopied` 어디에도 선언되지 않았다.

**원인:**
개발 과정에서 DID 문서 ID 복사 기능을 추가하다가 state 선언(`useState`) 및 데이터 바인딩 없이 함수만 남겨진 것으로 보인다.

**결과:**
- `copyDID()` 호출 시 즉시 `ReferenceError: docId is not defined` 발생
- 현재 렌더링 JSX에서 `copyDID`를 직접 호출하는 버튼이 없어 앱이 크래시되지는 않지만, 향후 버튼 연결 시 즉시 오류

**해결방안:**
```js
// 컴포넌트 상단에 추가
const [copied, setCopied] = useState(false)
// docId는 백엔드에서 받아오거나 localStorage에서 읽어야 함
const docId = localStorage.getItem('did_id') || ''
```
또는 현재 앱에서 DID 복사 기능이 불필요하다면 `copyDID` 함수 자체를 제거한다.

### [ENV-1] privy-server.mjs의 .env 로딩 경로 오류

**파일:** `privy-server.mjs:17`

**실제 코드:**
```js
const __dirname = dirname(fileURLToPath(import.meta.url))
dotenv.config({ path: join(__dirname, '../.env') })
```

**원인:**
`import.meta.url`이 `renter/privy-server.mjs`를 가리키므로 `__dirname`은 `renter/` 폴더다. 여기에 `../`를 붙이면 **프로젝트 루트**의 `.env`를 읽으려 한다.

실제 환경변수 파일은 `renter/.env`에 있다. (또는 없다면 더 큰 문제)

**결과:**
- `PRIVY_APP_SECRET`이 `undefined`로 로드됨
- `new PrivyClient({ appSecret: undefined })` → Privy SDK 초기화 실패 또는 API 호출 시 401/500 오류
- 회원가입 시 지갑 생성 불가

**해결방안:**
```js
// '../.env' → '.env' 로 수정
dotenv.config({ path: join(__dirname, '.env') })
```

### [ENV-2] main.jsx의 Privy appId 하드코딩

**파일:** `src/main.jsx:11`

**실제 코드:**
```jsx
<PrivyProvider
  appId="cmmtxys5k00a50djyxhziw4xs"   // 하드코딩
  ...
>
```

**원인:**
환경변수(`VITE_PRIVY_APP_ID`)를 사용하지 않고 appId를 직접 하드코딩했다.
`privy-server.mjs`는 `process.env.VITE_PRIVY_APP_ID`를 사용하는데, 두 값이 일치하지 않으면 클라이언트(React)와 서버(Node)가 서로 다른 Privy 앱을 바라보게 된다.

**결과:**
- 클라이언트와 서버가 다른 Privy 앱을 사용하면 지갑 주소 불일치 또는 인증 오류 발생
- 코드에 API 키가 노출됨 (보안 위험)

**해결방안:**
```jsx
// main.jsx
<PrivyProvider
  appId={import.meta.env.VITE_PRIVY_APP_ID}
  ...
>
```
그리고 `renter/.env`에 올바른 값으로 `VITE_PRIVY_APP_ID`를 통일한다.

### [CONFIG-1] vite.config.js 프록시 규칙 순서 충돌 (dead rule)

**파일:** `vite.config.js:42-54`

**실제 코드:**
```js
proxy: {
  '/ai': {           // ← 1순위로 매칭됨
    target: 'http://localhost:8000',
    rewrite: (path) => path.replace(/^\/ai/, '')
  },
  '/api': {
    target: 'http://localhost:8080',
  },
  '/ai/api': {       // ← 이 규칙은 절대 실행되지 않음
    target: 'http://localhost:8000',
    rewrite: (path) => path.replace(/^\/ai/, '')
  }
}
```

**원인:**
Vite의 프록시는 객체 키 선언 순서대로 매칭한다. `/ai/api/...`로 시작하는 URL은 더 구체적인 3번째 규칙 전에 이미 1번째 `/ai` 규칙에 걸린다. 3번째 규칙은 dead code다.

**결과:**
현재는 rewrite 규칙이 동일하므로 동작 자체에는 문제없다. 그러나 두 규칙이 달라지는 경우 혼란을 야기할 수 있다. (예: `/ai/api`에만 인증 헤더를 추가하려 할 때)

**해결방안:**
```js
proxy: {
  '/ai': {
    target: 'http://localhost:8000',
    changeOrigin: true,
    rewrite: (path) => path.replace(/^\/ai/, '')
  },
  '/api': {
    target: 'http://localhost:8080',
    changeOrigin: true
  }
  // '/ai/api' 규칙 삭제
}
```

### [MISC-1] CARE 토큰 컨트랙트 주소 미설정 (zero address)

**파일:** `src/utils/careToken.js:4`

// TODO: 실제 배포된 CareToken 컨트랙트 주소로 변경

### [API-1] renterLanguage() 함수 파라미터 누락

**파일:** `src/api/auth.js:100-101`

**실제 코드:**
```js
export const renterLanguage = () =>
  api.put('/api/renters/me/language').then((r) => r.data)
```

**원인:**
언어 코드를 body에 담아 PUT 요청을 보내야 하는데, 함수가 파라미터를 전혀 받지 않아 빈 body로 요청한다.

**결과:**
백엔드에서 body의 languageCode가 없어 400 오류 또는 null 처리될 것. 현재 이 함수를 실제로 호출하는 코드가 없어 당장의 오류는 없지만, 언어 설정 기능 자체가 동작하지 않는다.

**해결방안:**
```js
export const renterLanguage = (languageCode) =>
  api.put('/api/renters/me/language', { languageCode }).then((r) => r.data)
```

### [API-2] CarCrackPage의 AI 분석 엔드포인트 불일치 + mock 항상 실행

**파일:** `src/pages/car-crack/CarCrackPage.jsx:38`

**실제 코드:**
```js
const res = await fetch(`${AI_BASE}/analyze-damage`, { method: 'POST', body: form })
if (!res.ok) throw new Error('AI 서버 오류')
return res.json()
} catch {
// mock: 랜덤으로 1~2개 크랙 반환  ← 항상 여기로 fallback
```

**원인:**
실제 AI 서버의 스크래치 감지 엔드포인트는 `/api/v1/scratches/detect` (또는 유사)인데, 여기서는 `/analyze-damage`를 호출한다. 엔드포인트가 없으므로 404 → catch → mock 데이터 반환.

**결과:**
- 차량 반납 시 표시되는 스크래치 결과가 항상 랜덤 mock 데이터
- 실제 AI 분석이 수행되지 않음
- 블록체인 hash도 mock 고정값 `'0x34e...698d'` 기록

**해결방안:**
AI 서버의 실제 엔드포인트를 확인하고 URL을 수정한다. mock fallback은 개발 중에만 사용하고 프로덕션에서는 제거하거나 명시적으로 비활성화해야 한다.
