# 0. 홍채 인식 적용 가능성 검토

## 적용 대상

- **렌트카 & 응급환자**
  - 홍채 인식 아이디어 적용 가능
  - 렌트카 서비스에서는 필수 요소는 아님
  - 응급 환자 서비스에서는 핵심 인증 수단이 될 가능성 있음

## 기술적 구현 방식

- 정식 홍채 인식 시스템에는 적외선(IR) 센서가 필요함
- 본 프로젝트에서는 하드웨어 제약으로 인해 **갤럭시 스마트폰 카메라 기반 구현**을 가정
- 시연 목적:
  - 개인 홍채 데이터셋을 활용하여 파인튜닝 방식으로 구현
  - 과거 강의 자료에서 제공된 **CASIA-Iris 데이터셋** 활용 가능성 검토
- 실시간 인식이 아닌 **사진 입력 기반 테스트 코드 방식**으로 구현 예정
  - 기존 선배 개발 코드 참고 가능

# 1. AI 에이전트 DAO

## 1-1. 설득력

### 프로젝트 필요성

- AI 에이전트가 업무를 분담하는 시대적 흐름 반영
- 각 AI 에이전트의 수행 능력을 평가하고 관리하는 시스템 필요성 제시

### 블록체인 필요성

- DAO 거버넌스를 통해 평판 기반 권한 차등 부여
- 토큰 이코노미 설계 가능
- 다만, 발표 시 블록체인 필요성에 대한 설명이 다소 어려울 수 있음

## 1-2. 구현 가능성

- 에이전트 구축 + DAO 시스템 + 평판 점수 설계까지 모두 구현 시 난이도 매우 높음
- 현실적 대안:
  - LLM API 기반 간단한 에이전트 구성
  - DAO 기능은 최소화하여 구현

# 2. 국제 렌트카 서비스

## 2-1. 설득력

### 프로젝트 필요성

- 차량 반납 시 과도한 흠집 비용 청구 등 소비자 피해 예방
- 해외 렌트 시 현장 서류 작업 간소화
  - 운전면허증
  - 여권
  - 신용카드 정보 등 사전 인증

### 블록체인 필요성

- 차량을 NFT로 발행
- 차량 상태 이력 및 흠집 기록을 온체인에 저장
- 다음 이용자가 이전 기록을 조회 가능하도록 하여 투명성 확보

## 2-2. 구현 가능성

### 주요 구현 요소

- 사용자 DID (렌트카 업체 / 이용자)
- 차량 NFT 발행 및 흠집 보고서 로그 관리
- 스테이블 코인 결제 시스템
- 사진 기반 흠집 자동 인식 업로드
- 업체–사용자 간 실시간 채팅 및 번역
- 시연 확장 시 AIoT 기반 차량 제어 연동 가능성

※ 홍채 인식은 필수 요소는 아님

### 이미지 온체인 저장 문제

- 직접 온체인 저장은 현실적으로 어려움
- IPFS 기반 파일 저장 고려
  - 예: Pinata 등 IPFS 관리 서비스
- IPFS는 무결성은 보장하나 영속성은 별도 관리 필요

→ 전체 기능을 모두 구현하기에는 범위가 상당히 큼

# 3. 프롬프트 NFT 마켓

## 3-1. 설득력

### 프로젝트 필요성

- 생성형 AI 시대 도래
- 프롬프트의 지식 재산권 보호 어려움

### 블록체인 필요성

- 프롬프트 창작자의 소유권 증명
- 2차 판매 시 로열티 자동 분배
- 중앙 플랫폼 수수료 제거 가능

※ 단순 TXT 기반 프롬프트 마켓만 구현할 경우  
차별 요소가 로열티 제도 및 탈중앙 구조에 한정됨  
→ 설득력 확보 필요

## 3-2. 구현 가능성

- 서비스 내부에서 프롬프트 실행 및 결과물 검증 시스템 구현은 난이도 높음
- 별도 차별화 전략 필요

# 4. 응급 환자 DID 시스템

## 4-1. 설득력

### 프로젝트 필요성

- 응급 상황에서 보호자 부재 시
  - 기저질환
  - 알레르기
  - 복용 약물 정보 확인 불가 문제 해결

### 블록체인 필요성

- DID 기반 신원 인증
- 스마트 컨트랙트를 통해 조건부 의료 정보 공개
  - 병원 인증 시
  - 필요한 정보만 선택적 열람

## 4-2. 구현 가능성

### 홍채 인식

- 강력한 본인 인증 수단으로 활용 가능
- 사실상 필수 요소에 가까움

### 접근 제어 구조

- 환자 DID는 기본적으로 개인 Private Key로 보호
- 인증된 의료기관이 기관 지갑 주소 또는 인증서 기반 접근 요청
- 환자가 사전 설정에서:
  - “응급 시 인증된 의료기관에 정보 공개 허용” 옵션 활성화 시
  - 스마트 컨트랙트가 자동 승인

※ 법적 문제는 고려 대상에서 제외

### 시연 구성안

- 환자
  - DID 생성
  - 의료 데이터 업로드
  - 응급 공개 권한 설정
- 병원
  - 병원 인증 NFT 배지 발급
  - 응급 환자 DID 조회

※ 단순 DID 기능만으로는 기능이 제한적으로 보일 수 있음  
→ 의료 플랫폼적 요소 추가 검토 필요

# 5. 소아 환자 DAO

## 5-1. 설득력

### 프로젝트 필요성

- 병원 생활로 고립된 소아 환자들에게 정서적 교류 기회 제공
- 감성적 접근 가능

### 블록체인 필요성

- 현재 아이디어 수준에서는 블록체인 필수성이 낮음

## 5-2. 구현 가능성

- 소아 환자 인증 방식 설계 필요
- 어린이 친화적 UX 설계 필요
  - 가입
  - 인증
  - 사용 절차 전반 간소화
