## 2. Backend 설정 및 실행

### 2.1 환경 변수 설정

Backend는 `spring-dotenv` 라이브러리를 사용하여 `.env` 파일로부터 환경 변수를 로드합니다.

`backend/.env` 파일을 생성하고 다음 내용을 작성하세요:

```bash
# ─────────────────────────────────────────────
# Database
# ─────────────────────────────────────────────
DB_HOST=localhost
DB_PORT=3306
DB_NAME=care
DB_USER=root
DB_PASS=your_password

# ─────────────────────────────────────────────
# Redis
# ─────────────────────────────────────────────
REDIS_HOST=localhost
REDIS_PORT=6379

# ─────────────────────────────────────────────
# JWT
# ─────────────────────────────────────────────
JWT_SECRET=your_base64_encoded_secret_key

# ─────────────────────────────────────────────
# AWS S3 & CloudFront
# ─────────────────────────────────────────────
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
AWS_REGION=ap-northeast-2
S3_BUCKET_NAME=your_bucket_name
CLOUDFRONT_DOMAIN=your_cloudfront_domain

# ─────────────────────────────────────────────
# Blockchain (Polygon Amoy)
# ─────────────────────────────────────────────
RPC_URL=https://rpc.ssafy-blockchain.com
PRIVATE_KEY=your_wallet_private_key
CHAIN_ID=31221
CAR_NFT_ADDRESS=0x...
DID_REGISTRY_ADDRESS=0x...
MOCK_USDC_ADDRESS=0x...
DISPUTE_SETTLEMENT_ADDRESS=0x...
RENTAL_CONTRACT_ADDRESS=0x...

# ─────────────────────────────────────────────
# Privy (지갑 연동)
# ─────────────────────────────────────────────
PRIVY_APP_ID=your_privy_app_id
PRIVY_APP_SECRET=your_privy_app_secret
PRIVY_SERVER_URL=http://localhost:3001

# ─────────────────────────────────────────────
# Pinata (IPFS)
# ─────────────────────────────────────────────
PINATA_API_KEY=your_pinata_api_key
PINATA_API_SECRET=your_pinata_api_secret
PINATA_JWT=your_pinata_jwt

# ─────────────────────────────────────────────
# AI 서버 URL
# ─────────────────────────────────────────────
AI_YOLO_URL=http://localhost:8000
AI_FACE_URL=http://localhost:8001

# ─────────────────────────────────────────────
# 외부 API
# ─────────────────────────────────────────────
APICK_API_KEY=your_apick_api_key
NTS_API_KEY=your_nts_api_key
GMS_KEY=your_gms_key

# ─────────────────────────────────────────────
# JPA 설정 (선택)
# ─────────────────────────────────────────────
JPA_DDL_AUTO=update
JPA_SHOW_SQL=true

# ─────────────────────────────────────────────
# Context Path (선택)
# ─────────────────────────────────────────────
CONTEXT_PATH=/api
```

**프로덕션 환경 (Docker)**:
Docker 네트워크를 사용하는 경우 호스트명을 컨테이너 이름으로 변경
- `DB_HOST=mysql`
- `REDIS_HOST=redis`
- `PRIVY_SERVER_URL=http://privy-server:3001`
- `AI_YOLO_URL=http://<GPU 서버 주소>:8000` (외부 GPU 서버)
- `AI_FACE_URL=http://ai-verify:8001`

### 3.1 환경 변수 설정

#### 개발 환경 (.env.local)

`renter/.env.local` 파일을 생성하고 다음 내용을 작성하세요:

```bash
VITE_API_BASE_URL=http://localhost:8080
VITE_AI_WS_URL=ws://localhost:8000
VITE_PRIVY_SERVER_URL=http://localhost:3001
```

#### 프로덕션 환경 (.env.production)

`renter/.env.production` 파일 (Jenkins Credentials로 주입됨):

```bash
VITE_API_BASE_URL=https://j14e201.p.ssafy.io
VITE_AI_WS_URL=wss://j14e201.p.ssafy.io/ai
VITE_PRIVY_SERVER_URL=https://j14e201.p.ssafy.io
```

> **주의**: HTTPS 환경에서는 WebSocket도 `wss://`를 사용해야 Mixed Content 에러가 발생하지 않습니다.
> 코드에서는 `window.location.protocol === 'https:'` 감지 시 자동으로 Nginx 프록시 경로(`wss://도메인/ai/api/v1/scratches/ws/detect`)를 사용합니다.

### 3.2 Privy 보조 서버 환경 변수

`renter/.env` (privy-server.mjs에서 사용):

#### 환경 변수 설정

`ai/yolo/.env` 파일을 생성하세요:

```bash
AWS_REGION=ap-northeast-2
S3_BUCKET_NAME=your_bucket_name
CLOUDFRONT_DOMAIN=your_cloudfront_domain

# Pinata (IPFS proof)
PINATA_JWT=your_pinata_jwt
```

#### Docker로 실행 (GPU 서버)

```bash
# 이미지 빌드
docker build -t ai-yolo:latest ai/yolo/

# GPU 컨테이너 실행
docker run -d \
  --name ai-yolo \

  --env-file ai/yolo/.env \

  --restart unless-stopped \
  ai-yolo:latest
```

```bash
cd ai/yolo
```

GPU 서버가 실행 중인 URL을 Backend `.env`의 `AI_YOLO_URL`에 설정합니다:

```bash
# 예시 (cloudflare tunnel 사용 시)

# 예시 (직접 IP 사용 시)
AI_YOLO_URL=http://213.173.99.23:16244
```

Jenkins 배포 시 `AI_YOLO_URL` 환경 변수를 Jenkins 시스템 설정에서 지정해야 합니다.

## 6. Nginx 설정 및 실행

### 6.1 설정 파일

`infra/nginx/nginx.conf` 파일이 기본 설정 파일입니다.

주요 설정 내용:
- HTTP → HTTPS 리다이렉션
- SSL/TLS 인증서 설정 (Let's Encrypt, `j14e201.p.ssafy.io`)
- Renter 앱 정적 파일 서빙 (`/`, Blue-Green 배포)
- Company 앱 정적 파일 서빙 (`/company/`, Blue-Green 배포)
- Backend API 리버스 프록시 (`/api/`) — SSE 지원
- AI-Verify 프록시 (`/ai/api/v1/face/`, `/ai/api/v1/ocr/`, `/ai/api/v1/scratches/compare`)
- YOLO WebSocket 프록시 (`/ai/api/v1/scratches/ws/`) — 외부 GPU 서버
- Privy 서버 프록시 (`/privy/`)
- Jenkins 프록시 (`/jenkins/`)
- 헬스체크 엔드포인트 (`/health`)

**Blue-Green Placeholder**:
- `__RENTER_ROOT__`: Renter 빌드 파일 경로 (Jenkins에서 치환)
- `__COMPANY_ROOT__`: Company 빌드 파일 경로 (Jenkins에서 치환)
- `__BACKEND_HOST__`: 활성 Backend 컨테이너 이름 (Jenkins에서 치환)
- `__AI_YOLO_URL__`: GPU 서버 URL (Jenkins 환경 변수에서 치환)

### 6.2 SSL 인증서 설정

Let's Encrypt를 사용하여 SSL 인증서를 발급받습니다:

```bash
sudo apt-get update
sudo apt-get install certbot python3-certbot-nginx

# 인증서 발급
sudo certbot certonly --nginx -d j14e201.p.ssafy.io
```

인증서는 다음 경로에 저장됩니다:
- 인증서: `/etc/letsencrypt/live/j14e201.p.ssafy.io/fullchain.pem`
- 개인키: `/etc/letsencrypt/live/j14e201.p.ssafy.io/privkey.pem`

### 6.3 Nginx 컨테이너 실행

프로덕션에서는 `docker-compose.yml`로 실행합니다 (섹션 8 참조).

수동으로 실행하는 경우:

```bash
docker run -d \
  --name nginx \
  --network care-network \
  -p 80:80 \
  -p 443:443 \
  -v /home/ubuntu/infra/nginx/nginx.conf:/etc/nginx/conf.d/default.conf:ro \
```

## 11. 트러블슈팅

### 11.1 Backend 관련

**문제: 데이터베이스 연결 실패**
```
Cannot create PoolableConnectionFactory
```
해결방법:
1. MySQL 컨테이너 실행 여부 확인: `docker ps | grep mysql`
2. `.env` 파일의 `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASS` 확인
3. Docker 환경에서는 `DB_HOST=mysql` 로 설정

**문제: Redis 연결 실패**
```
Unable to connect to Redis
```
해결방법:
1. Redis 컨테이너 실행 여부 확인: `docker ps | grep redis`
2. Docker 환경에서는 `REDIS_HOST=redis` 로 설정

**문제: Blockchain RPC 연결 실패**

해결방법:
1. `RPC_URL`이 올바른지 확인 (`https://rpc.ssafy-blockchain.com`)
2. `PRIVATE_KEY`, 컨트랙트 주소들이 올바르게 설정되었는지 확인
3. 네트워크 `CHAIN_ID`가 `31221`인지 확인

### 11.2 Frontend 관련

**문제: API 호출 시 Network Error**

해결방법:
1. `.env.production`의 `VITE_API_BASE_URL`이 올바른지 확인
2. 빌드 후 환경 변수가 번들에 반영되었는지 확인 (빈 값이면 `undefined` 처리됨)
3. Jenkins Credentials의 `renter-env-file` 내용이 올바른지 확인

**문제: WebSocket Mixed Content 에러**
```
Mixed Content: ... wss:// ... blocked
```
해결방법:
1. `VITE_AI_WS_URL`이 `wss://`로 시작하는지 확인
2. HTTPS 환경에서는 코드가 자동으로 Nginx 프록시 경로를 사용하므로 별도 처리 불필요
3. Nginx의 `/ai/api/v1/scratches/ws/` 경로 설정 확인

**문제: Company 앱 정적 파일 로드 실패 (404)**

해결방법:
1. `vite.config.js`의 `base: '/company/'` 설정 확인
2. Nginx 설정에서 `/company/` alias가 올바르게 설정되었는지 확인
3. `COMPANY_ROOT`가 올바른 디렉토리를 가리키는지 확인

**문제: 의존성 설치 실패**
```
npm ERR! code ERESOLVE
```
해결방법:
```bash
rm -rf node_modules package-lock.json
npm install
# 또는
npm install --legacy-peer-deps
```

### 11.3 AI 서버 관련

**문제: AI-Verify 컨테이너 시작 실패 (DeepFace 모델 다운로드 오류)**

해결방법:
1. 빌드 시 네트워크 연결 확인 (HuggingFace에서 모델 다운로드)
2. 이미 다운로드된 경우 `/root/.deepface/weights/facenet512_weights.h5` 존재 확인

**문제: YOLO 서버 WebSocket 연결 불가**

해결방법:
1. `AI_YOLO_URL`이 현재 GPU 서버 URL로 설정되었는지 확인
2. Nginx의 `__AI_YOLO_URL__` placeholder가 치환되었는지 확인:
   ```bash
   grep AI_YOLO /home/ubuntu/infra/nginx/nginx.conf
   ```
3. GPU 서버가 실행 중인지 확인

**문제: S3 업로드 실패**

해결방법:
1. `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`가 올바른지 확인
2. S3 버킷 정책에서 해당 IAM 사용자에게 PutObject 권한이 있는지 확인
3. `S3_BUCKET_NAME`, `AWS_REGION`이 올바른지 확인

### 11.4 Docker 관련

**문제: care-network not found**
```
network care-network not found
```
해결방법:
```bash
docker network create care-network
```

**문제: 컨테이너 간 통신 실패**

해결방법:
1. 모든 컨테이너가 `care-network`에 연결되어 있는지 확인
2. `docker network inspect care-network`로 연결된 컨테이너 목록 확인

### 11.5 Jenkins 관련

**문제: Docker 명령어 실행 실패 (permission denied)**
```
permission denied while trying to connect to the Docker daemon socket
```
해결방법:
```bash
sudo usermod -aG docker jenkins
sudo systemctl restart jenkins
# 또는 Jenkins 컨테이너 재시작
docker restart jenkins
```

**문제: Credentials를 찾을 수 없음**
```
No such credentials
```
해결방법:
1. `backend-env-file`, `renter-env-file`, `ai-verify-env-file` ID가 정확히 일치하는지 확인
2. Credentials가 Global scope로 설정되어 있는지 확인

**문제: Nginx reload 실패**

해결방법:
1. nginx.conf 문법 오류 확인: `docker exec nginx nginx -t`
2. placeholder가 모두 치환되었는지 확인 (`__`로 검색)
3. Backend/Renter/Company 디렉토리의 `active_color` 파일이 존재하는지 확인

### 11.6 Nginx 관련

**문제: 502 Bad Gateway**

해결방법:
1. Backend 컨테이너가 실행 중인지 확인: `docker ps | grep backend`
2. Nginx의 `resolver 127.0.0.11` 설정으로 컨테이너 DNS 조회 확인
3. `__BACKEND_HOST__`가 올바른 컨테이너명으로 치환되었는지 확인

**문제: SSE (Server-Sent Events) 연결이 끊어짐**

해결방법:
1. Nginx의 `proxy_read_timeout 86400s` 설정 확인
2. `proxy_buffering off`, `proxy_cache off` 설정 확인
3. 중간 프록시나 로드밸런서가 연결을 끊지 않는지 확인
